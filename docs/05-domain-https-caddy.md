# 05 — Domain, DNS, Caddy and HTTPS

## Purpose

This document explains how the project was exposed publicly with a real domain and HTTPS.

Final public URL:

```text
https://pokedex.badiscloud.fr
```

---

## Final traffic flow

```text
User browser
→ https://pokedex.badiscloud.fr
→ OVH DNS
→ Elastic IP AWS
→ EC2 security group 443
→ Caddy container
→ Docker network
→ Node.js app container
```

---

## Domain acquisition

Domain chosen:

```text
badiscloud.fr
```

Purchased from OVH.

Only the domain was needed.

Not needed:

- OVH web hosting
- WordPress hosting
- OVH paid SSL
- OVH mail
- CDN
- extra web packages

The application is hosted on AWS, not OVH.

OVH is used only for DNS/domain registration.

---

## DNS record

OVH DNS record:

```text
Type: A
Subdomain: pokedex
Target: <ELASTIC_IP>
```

This creates:

```text
pokedex.badiscloud.fr → <ELASTIC_IP>
```

---

## DNS propagation

OVH says propagation can take up to 24 hours.

In practice, a new record often works within minutes.

Verification command:

```bash
dig pokedex.badiscloud.fr A
```

Expected answer:

```text
pokedex.badiscloud.fr. 3600 IN A <ELASTIC_IP>
```

Test public resolvers:

```bash
dig @1.1.1.1 pokedex.badiscloud.fr A
dig @8.8.8.8 pokedex.badiscloud.fr A
```

---

## Why Elastic IP is needed

Without Elastic IP, AWS assigns a temporary public IP to the EC2 instance.

After each destroy/apply, the IP changes.

DNS would break.

Elastic IP gives a stable IP:

```text
<ELASTIC_IP>
```

DNS can safely point to it.

---

## Elastic IP lifecycle

The Elastic IP was created manually and reused by Terraform through its allocation ID.

Reason:

```text
EC2 should be temporary.
Elastic IP should stay stable.
DNS should not change.
```

Tradeoff:

```text
Elastic IP has an hourly public IPv4 cost.
```

---

## Security group for HTTPS

Terraform opens:

```text
80/tcp
443/tcp
```

Why both?

| Port | Purpose |
|---|---|
| 80 | HTTP, Let's Encrypt validation, redirect |
| 443 | HTTPS traffic |

Port 22 remains closed.

---

## Why Caddy?

Caddy was chosen because it provides:

- simple reverse proxy config
- automatic HTTPS
- automatic Let's Encrypt certificate management
- automatic HTTP to HTTPS behavior
- simple Docker usage

---

## First Caddy attempt

The first attempt tried to install Caddy through a DNF COPR repository:

```bash
dnf copr enable -y @caddy/caddy
```

This failed on Amazon Linux 2023:

```text
Repository 'amazonlinux-2023-x86_64' does not exist in project '@caddy/caddy'
```

So Caddy was switched to Docker.

---

## Final Caddy approach

Caddy runs as a Docker container:

```bash
docker run -d \
  --name caddy \
  --network pokedex-network \
  --restart unless-stopped \
  -p 80:80 \
  -p 443:443 \
  -v /opt/caddy/Caddyfile:/etc/caddy/Caddyfile:ro \
  -v /opt/caddy/data:/data \
  -v /opt/caddy/config:/config \
  caddy:2
```

---

## Caddyfile

Generated by Ansible:

```text
pokedex.badiscloud.fr {
    reverse_proxy pokedex-ci-cd-lab:3000
}
```

This means:

```text
For requests to pokedex.badiscloud.fr,
proxy traffic to the app container on port 3000.
```

---

## Why Docker network?

Caddy and the app run in the same Docker network:

```text
pokedex-network
```

This allows Caddy to reach the app by container name:

```text
pokedex-ci-cd-lab:3000
```

The app does not need to publish port 3000 to the EC2 host.

---

## 502 troubleshooting

At one point, HTTPS returned:

```text
HTTP/2 502
server: Caddy
```

Meaning:

```text
Browser → Caddy = OK
Caddy → Node app = KO
```

The issue was reverse proxy connectivity.

Fix:

```text
Put Caddy and app in the same Docker network.
Use reverse_proxy pokedex-ci-cd-lab:3000.
```

---

## Why not expose Node directly?

Before Caddy:

```text
Internet → EC2:80 → Node container
```

After Caddy:

```text
Internet → EC2:443 → Caddy → Node container
```

Only Caddy is public.

Node remains internal.

This is better because:

- TLS terminates at Caddy
- Node app stays private
- one public entry point
- cleaner production-style architecture

## Certificate storage

Caddy stores certificate data in:

```text
/opt/caddy/data
```

Mounted into the container:

```text
-v /opt/caddy/data:/data
```

If the EC2 is destroyed, certificate data is lost.

That is acceptable because Caddy can request a new certificate on redeploy.

Be careful with Let's Encrypt rate limits if repeatedly recreating many certificates quickly.
